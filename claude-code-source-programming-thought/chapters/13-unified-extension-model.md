# CH13 命令、技能、插件：统一扩展模型

状态：`completed`

## 一句话摘要

Claude Code 并不是分别维护命令、技能和插件三套完全独立的扩展体系，而是把它们尽量折叠到统一的 `Command` 协议与同一条能力装配链上。

## 问题引入

扩展机制几乎总是 Agent 系统走向复杂化的第一现场。最初只有几个内建命令时，所有能力都可以直接硬编码；后来有了技能、插件、工作流、命令来源标签和模型可见性控制，系统很容易滑向三套甚至更多平行机制。用户会开始区分“这个是 slash command”“那个是 skill”“还有一个是 plugin command”，而源码也会随之分裂成多个无法统一治理的入口。

Claude Code 在这里采取了一个非常鲜明的立场：不管扩展来自哪里，只要它最终要成为模型或用户可以调用的能力，它就应该尽量落到统一协议上。这并不意味着所有扩展在来源和生命周期上完全一样，而是意味着它们在“如何被发现、筛选、暴露和调用”这件事上应该共享同一套骨架。

因此，本章要解释的不是“命令、技能、插件分别是什么”，而是“Claude Code 如何把不同形态的扩展折叠成统一能力面”。这个问题回答清楚后，你才能理解为什么插件更像装配容器，而不是第三套执行引擎。

## 思想命题

本章的核心命题是：Claude Code 的统一扩展模型，关键不在于消灭命令、技能和插件之间的差异，而在于让这些差异停留在来源层，而不是蔓延到运行时接口层。

第一，`Command` 是共同协议。无论能力来自内建命令、技能目录还是插件加载，只要它最终进入可调用集合，就会尽量投影成 `types/command.ts` 描述的命令对象。

第二，插件是装配容器，不是平行运行时。插件可以贡献命令、技能、配置和其他能力来源，但它最终仍然要经过命令装配与过滤流程才能被系统消费。

第三，技能不是独立引擎，而是 prompt 型命令在语义上的一种特殊视图。它们通常带 `whenToUse`、来源标签和模型调用限制，但在更深层的能力协议上仍与命令共享骨架。

第四，统一扩展模型的真正价值在治理。只有当不同来源的能力最终进入同一协议，系统才能统一处理 availability、isEnabled、loadedFrom、disableModelInvocation、上下文模式与模型可见性。

## 源码剖面

这一章最重要的入口文件是 [commands.ts](/Users/lizixuan/Documents/IdeaProjects/cc-source/src/commands.ts)。它并不是简单列出所有 slash command，而是承担了“能力总装配器”的角色。Claude Code 会在这里汇总内建命令、工作流命令、技能目录产物和插件贡献，再依据当前环境、配置和可见性策略过滤出最终集合。这个入口本身已经说明：系统关心的是统一装配后的能力面，而不是各来源各说各话。

[types/command.ts](/Users/lizixuan/Documents/IdeaProjects/cc-source/src/types/command.ts) 则给出了统一协议。这里至少可以看到 `prompt`、`local`、`local-jsx` 等命令形态，也能看到 `context`、`disableModelInvocation`、元数据来源等字段。这个定义非常关键，因为它意味着 Claude Code 不要求所有扩展“长得一样”，但要求它们都能被投影成一套可治理的命令模型。换句话说，差异被允许存在于实现层，而不是暴露层。

技能系统的核心在 [skills/loadSkillsDir.ts](/Users/lizixuan/Documents/IdeaProjects/cc-source/src/skills/loadSkillsDir.ts)。这个文件做的并不只是读取 Markdown 技能文档，更重要的是把技能解析成可被系统理解的结构，包括名称、描述、`whenToUse`、路径触发、忽略规则以及是否允许模型直接调用等信息。站在统一扩展模型的视角看，技能的本质不是“另一类神秘对象”，而是带更强语义标签的 prompt 型能力条目。

插件相关逻辑集中在 [utils/plugins/pluginLoader.ts](/Users/lizixuan/Documents/IdeaProjects/cc-source/src/utils/plugins/pluginLoader.ts) 与 [utils/plugins/loadPluginCommands.ts](/Users/lizixuan/Documents/IdeaProjects/cc-source/src/utils/plugins/loadPluginCommands.ts)。这里可以看到 Claude Code 并不让插件绕开命令系统直接进入运行时，而是要求插件先把自己能贡献的命令显式交出来，再并入统一装配流程。这个细节非常重要，因为它说明插件被设计成“能力来源容器”，而不是“拥有特权通道的扩展沙箱”。

进一步看 [commands.ts](/Users/lizixuan/Documents/IdeaProjects/cc-source/src/commands.ts) 的 `getCommands()`、`loadAllCommands()` 一类入口，就能发现 Claude Code 在这里只做一件事：把能力收束为统一列表，再基于 availability、enabled 状态、来源标签和上下文模式作筛选。也就是说，不同扩展真正的区别主要体现在“怎么来”和“何时可见”，而不是“被调用时走不走同一条路”。

这一点在“技能视图”上尤其明显。技能经常被用户理解为“知识包”或“工作方式”，但在 Claude Code 源码里，它们仍要在很大程度上服从命令协议。技能之所以看起来不同，是因为它们往往更强调 `whenToUse`、文档语义和模型调用限制，而不是因为它们有完全不同的执行底座。

因此，本章可以还原出这样一条主链：`commands.ts` 汇总内建、技能、插件和工作流来源，`types/command.ts` 提供统一命令协议，`loadSkillsDir.ts` 把技能投影为命令化结构，插件 loader 负责把插件贡献变成命令条目，最后 `getCommands()` 之类入口再筛出最终可见能力面。

## 设计取舍

Claude Code 在这一章的第一个取舍，是概念纯粹性与治理统一性之间的平衡。完全可以为技能、插件和命令各设计一套“最适合自己”的数据结构与调用入口，但那样做会让后续的过滤、权限、模型可见性和 UI 投影越来越难统一。Claude Code 明显更重视统一治理。

第二个取舍，是插件灵活性与系统边界之间的平衡。允许插件自带一切运行时特权，看似更开放；但那会迅速破坏整个系统的能力装配秩序。Claude Code 选择让插件作为容器参与统一流程，实际上是在牺牲部分自由度，换系统边界稳定。

第三个取舍，是技能语义与命令协议之间的平衡。技能确实比普通命令带有更强的知识与建议属性，但 Claude Code 仍然尽量把它们折叠到同一协议上。这样做会让“技能的独特性”在实现层看起来没那么浪漫，但换来的是运行时处理的一致性。

## 工程启示

如果你要设计自己的扩展系统，Claude Code 的经验非常直接。

第一，先定义统一能力协议，再接不同来源。不要让来源差异直接变成运行时分裂。

第二，把插件当作贡献能力的容器，而不是特权运行时。只有这样，系统才能保持整体可治理性。

第三，让技能、模板、工作流之类更高语义的扩展，尽量投影到同一协议上。差异可以存在于元数据层，不必蔓延到底层 ABI。

第四，统一处理 availability、enabled、来源标签和模型可见性。扩展多了以后，治理成本往往比加载成本更高。

## 思考题

1. 如果让插件绕过 `Command` 协议直接把能力注入运行时，Claude Code 后续哪些治理逻辑会最先失去一致性？
2. 技能为什么适合被看成“带语义标签的 prompt 型命令”，而不是完全独立的对象类型？
3. 如果以后要增加第四种扩展形态，应该先扩展统一协议，还是先引入一套新运行时？为什么？

## 本章不覆盖什么

本章不展开每种命令类型的具体执行细节，不逐个分析插件 manifest 字段，也不替代后续 MCP 与外部边界章节。这里关注的是 Claude Code 如何把不同来源的扩展折叠成统一能力面。
