# 写作规范

## 目标

全书要解释“为什么这样设计”，而不只是“这里有什么文件”。每章都必须从思想命题出发，再回到源码实现和工程约束。

## 叙事结构

正文固定使用以下一级标题：

1. `问题引入`
2. `思想命题`
3. `源码剖面`
4. `设计取舍`
5. `工程启示`
6. `思考题`
7. `本章不覆盖什么`

## 行文要求

- 语言统一使用中文。
- 优先解释抽象边界、状态迁移、约束来源、失效模式。
- 不写“源码很复杂”“这里很多文件”这类空话。
- 不把目录结构误写成架构结论。
- 不根据函数名猜测行为，必须回到实现、注释或调用链。
- 允许引用源码文件路径，但不要大段粘贴代码。
- 每章至少写清一个明确的 trade-off。

## 《JAVA编程思想》式写法约束

- 先提出一个“思想命题”，再用源码论证它。
- 章节要有递进关系，后一章默认读者已经理解前一章的核心概念。
- 允许在“工程启示”中抽象出可迁移方法，但必须明确哪些结论来自 Claude Code 的特定约束。
- 不做手把手教程，不做 API 参考手册，不做逐文件摘要。

## 引用要求

- 每章至少 3 个源码锚点。
- 每章至少 1 条主调用链。
- 每章至少 1 个关键 trade-off。
- 每章必须写“本章不覆盖什么”。

## 多 Agent 协作要求

每章必须经过以下分工：

- `brainstorm`：冻结命题、读者收益、排除范围、关键源码。
- `evidence harvest`：列出章节事实基础和关键文件。
- `call-chain tracing`：还原至少 1 条主路径。
- `chapter draft`：把事实提升为思想叙事。
- `fact check`：逐段回看是否有无法落地到源码的句子。
- `polish and cross-link`：补和前后章节的概念链接。

## 状态定义

- `pending`：尚未进入写作。
- `in_progress`：已冻结命题，正在编写或校验。
- `completed`：正文与 evidence 都通过校验。
